## Roundcube Catch-all — Add-on Documentation

This add-on adds the **catch-all** Roundcube plugin
([`teh-hippo/roundcube-catchall`](https://github.com/teh-hippo/roundcube-catchall))
to the **Roundcube Webmail** add-on.

It is a companion to, not a replacement for, the Roundcube Webmail add-on.

### What it does

On startup this add-on:

1. Clones the plugin source at a **pinned commit** into a staging folder.
2. Writes `config.inc.php` alongside it with the options you configured.
3. Drops a `.ha-plugin-ready` marker and atomically swaps the staging folder
   into `/share/roundcube/plugins/roundcube_catchall/`.
4. Stays running so that any change to its options triggers a fast restart
   and re-provisions the plugin.

The **Roundcube Webmail** add-on scans `/share/roundcube/plugins/` on its own
startup and only loads entries that carry the readiness marker, so it never
sees a half-written tree. No rebuild needed.

### Configuration

Settings → Add-ons → Roundcube Catch-all → Configuration.

- **Pre-fill From on reply** — default on. Recommended.
- **Skip Roundcube login screen** — default off. See **Autologin** below.
- **Auto-login email / password** — required when autologin is on. Use the
  wildcard alias (`*@example.com`) or a specific alias, with an IMAP/SMTP
  password generated by your mail provider.

The catch-all domain is derived from the logged-in user's email address — no
separate domain option is needed.

### How it works at reply time

When you reply to a message whose recipient matches the catch-all domain, the
plugin pre-fills **From** with the original recipient address and creates a
local Roundcube identity. Outbound sending uses the logged-in SMTP session,
so your mail provider must accept arbitrary `From` on the domain (Forward
Email does when the password is generated for a wildcard alias).

Inbound still flows through the catch-all exactly as before.

No alias churn, no per-reply API calls, no stored per-identity credentials.

### Autologin

When enabled, the plugin injects a synthetic login POST on every anonymous
request, so the Roundcube login form is skipped. It only applies to the
single identity you configure.

**Security tradeoff**: anyone who can reach this Roundcube instance is
logged in as that alias. In the HA add-on deployment the gate is HA Ingress
authentication — so you are effectively saying "any HA user who can see the
sidebar is also this email user". Fine for a single-user household; avoid
if you share HA with family or guests.

The password is written into `config.inc.php` under
`/share/roundcube/plugins/roundcube_catchall/` with mode `0640`.

### Order of operations

1. Install the **Roundcube Webmail** add-on first and confirm it works.
2. Install **Roundcube Catch-all** alongside it.
3. Configure this add-on and start it.
4. Check its log for `Plugin provisioned` confirmation.
5. Restart the **Roundcube Webmail** add-on to pick up the new plugin.

### Troubleshooting

#### Plugin not picked up

Check that the Roundcube Webmail add-on has been restarted since this
add-on first started. The shared-folder symlink only resolves at
Roundcube's startup.

#### Outbound mail rejected with wildcard alias

Verify your provider accepts arbitrary `From` on an authenticated wildcard
session. With Forward Email this means generating the SMTP password against
the wildcard alias (`*@yourdomain`), not a specific alias.

### Data persistence

This add-on writes to `/share/roundcube/plugins/roundcube_catchall/` which
lives in the HA shared folder. Backups include `/share/` by default. The
add-on itself holds no state under `/data/`.
